# Project 1 Teach-back — Linux Server Administration

This is my study script for interviews about Project 1. Every section is the answer to a question an interviewer might ask, written in my own words.

## "Can you walk me through this Linux server project?"

I provisioned an Ubuntu 24.04 server on AWS Lightsail and hardened it: SSH key auth only, root login off, password auth off, cloud firewall locked to my home IP, fail2ban watching brute-force attempts. I wrote a backup script that tars `/etc`, `/home/ubuntu`, and `/var/log/nginx` into a date-stamped directory under `/var/backups/server/`, runs daily via a systemd timer, and prunes anything older than seven days. About three days in I noticed unfamiliar SSH sessions on the box, traced it back to a credential I had carelessly handed to a chat tool, and decided to rebuild from scratch rather than try to clean a possibly-compromised host. The project taught me Linux fundamentals — filesystem, permissions, systemd, logging, networking — and on top of that, that detection is mostly the same skill as operation, just used to check your assumptions about who else might be on the box.

## "What is the Linux filesystem layout?"

Linux follows the Filesystem Hierarchy Standard, which is the agreement that lets you walk onto any Linux box and know where things are without docs. `/etc` is system-wide config — sshd, nginx, fstab, fail2ban all live there. `/var/log` is logs — `auth.log` for SSH attempts, `syslog` for general system messages, plus whatever apps drop in. `/home/<user>` is per-user space. `/usr/local/bin` is for binaries you installed yourself, outside the package manager. `/tmp` is scratch space, cleared on reboot. The standardization is the whole point — without FHS, every distro would be a different machine to operate, and Linux skills wouldn't transfer.

## "Explain Linux permissions to me."

Every file has three permission buckets — owner, group, others — and each bucket has three actions: read, write, execute. In numeric notation: `r=4`, `w=2`, `x=1`, and you sum them per bucket. So `chmod 750 backup.sh` is owner `rwx` (4+2+1=7), group `r-x` (4+0+1=5), others nothing. On directories the execute bit means something subtle — it's the right to enter the directory and reach files inside, not the right to "run" the directory; without it, even files you have `r` on are unreachable. `chmod 777` is almost always wrong, because it gives every user on the system full control of the file; the only reason to do it is when you don't yet understand who needs what access, and the right move then is to figure that out, not to flatten everything.

## "What is systemd and how do you manage services with it?"

systemd is the init system on every modern Linux distro — it's PID 1, it starts everything else, and it represents every service, mount, socket, and scheduled job as a "unit". Unit files live in `/etc/systemd/system/` (your own) or `/lib/systemd/system/` (shipped by packages), and you control them with five commands: `systemctl start`, `stop`, `restart`, `enable`, `disable`. The important distinction is that `start/stop` change live state right now, while `enable/disable` change whether the unit comes up at boot — you can have a unit that's started-and-not-enabled (running now, gone after reboot) or enabled-but-not-started (will come up next boot, not currently running). For my backup, I have a `Type=oneshot` service that runs the script and exits, paired with a `.timer` unit that triggers it daily — that's how you do cron on modern Linux, with the upsides of structured journal logging, automatic catch-up via `Persistent=true`, and the script's exit code visible in `systemctl status`.

## "How do you read logs on a Linux server?"

Two log systems live on modern Linux. The traditional one is flat files in `/var/log` — `auth.log` for SSH, `syslog` for general messages, `nginx/access.log` and `error.log` for nginx. The newer one is the systemd journal, queried via `journalctl`. The journal is structured — every entry has metadata about who wrote it, when, and at what priority — so you can do narrow queries like `journalctl -u backup.service --since "1 hour ago" -p err` to get only errors from one unit in a specific window. For an unfamiliar machine I usually start with `journalctl -xe` to see recent errors with explanations, then narrow down by unit or time. The two systems overlap on purpose — sshd writes to both `auth.log` (so fail2ban can read it) and the journal (so I can query it structurally).

## "Walk me through the security on this server."

Seven overlapping controls. That's the defense-in-depth principle — no single failure is enough to compromise the box. **One**, the Lightsail cloud firewall restricts SSH to my home IP, so the port isn't even reachable from the rest of the internet. **Two**, SSH accepts key-based auth only — passwords are disabled outright in `sshd_config`. **Three**, root login is disabled, so even if someone gets the `ubuntu` key they can't pivot to root without sudo. **Four**, `MaxAuthTries 3` and `LoginGraceTime 30` tighten the SSH negotiation window so brute-forcers don't get many tries per connection. **Five**, fail2ban watches `/var/log/auth.log` and bans IPs that fail repeatedly. **Six**, `unattended-upgrades` patches security updates daily. **Seven**, everything that happens on the box lands in journalctl, so after an incident I can reconstruct who connected, when, from where. Peel off any one layer and the others still hold. That's the point of defense in depth — you assume any single control will fail eventually, and you design so that failure isn't catastrophic.

## "Tell me about a time something broke on this project."

Three days into setup I ran `ss -tn` to look at TCP connections and saw active sessions from IPs I didn't recognize. `who` showed `ubuntu` logged in from those IPs. `last -i` confirmed they'd been connecting for hours.

The root cause was on me. I had shared the `.pem` file with an AI assistant earlier that week while debugging a connection issue, treating it as if it were a config snippet rather than a credential. The lesson is simple — credentials never go into chat tools, not as text, not as attachments, not for "just a second" debugging.

I made the hard call: rebuild from scratch instead of trying to clean a possibly-compromised host. There's no command that confidently proves a Linux box is uncompromised after a real intrusion. So I deleted the instance, generated a new ed25519 key locally that has never left my machine, and rebuilt with the full hardening from day one rather than retrofitting it.

The deeper takeaway was about detection. The reason I noticed at all was that I had already learned `ss`, `who`, and `last` defensively — to verify my own connection during setup. The exact same commands that confirm "I'm connected" reveal "someone else is too." Defensive command-line literacy is detection. Every tool you learn for the happy path is also a tool for the unhappy path. That changed how I think about every command I'm taught now.

## "Tell me about your backup script."

It tars `/etc`, `/home/ubuntu`, and `/var/log/nginx` into a date-stamped directory under `/var/backups/server/YYYY-MM-DD/`, with one `.tar.gz` per source so I can extract just one source from a specific day without untarring the whole thing. It runs via a systemd timer set to `OnCalendar=daily` with `RandomizedDelaySec=15m` so the start time jitters within a 15-minute window — pointless on one host, important if I ever scale this to a fleet that would otherwise hammer something at the same second. Three engineering choices worth defending: **`set -euo pipefail`** at the top so the script fails on any error, undefined variable, or pipe failure instead of silently producing a bad archive; **a defensive guard** that checks each source directory exists before adding it to the `tar` arg list — handles the case where someone deletes a directory I expected; **running via systemd timer instead of cron** so I get structured journal logs, automatic catch-up via `Persistent=true` after a reboot, and the exit code visible in `systemctl status`. It also tees output to `/var/log/backup.log` so I have a flat-file fallback if the journal is rotated or unavailable.

## "What would you do differently if this were production?"

Several things. **Off-host backups.** Right now everything lands on the same disk as the server, so a disk failure loses both. Real production writes to S3 or another machine; the script is shaped so swapping the `tar` block for `aws s3 sync` would be a small change. **Infrastructure as code.** I provisioned via the Lightsail console — if the box dies I rebuild by clicking. Real production has the entire machine described in Terraform and configured by Ansible so it can be reproduced from git in minutes. **Active monitoring, not passive logs.** Right now I have to look at journalctl to know something's wrong; real production has Prometheus or CloudWatch scraping metrics with alert rules paging me when thresholds break. **Centralized log shipping.** Per-host `auth.log` and fail2ban don't scale; a SIEM ingesting from every host plus banning at the network edge does. **Real TLS, real domain.** Plain HTTP is fine for a learning environment; production needs a Let's Encrypt cert and a domain so traffic isn't readable on the wire.
